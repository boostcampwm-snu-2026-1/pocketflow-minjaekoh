# Pocketflow 최종 발표 영상 제출용 정리

이 문서는 `루카스 최종영상제출` 폼에 제출할 최종 발표 영상을 준비하기 위한 설명서입니다.

목표는 두 가지입니다.
- 영상에서 말할 순서를 그대로 따라갈 수 있게 구성한다.
- 프로젝트의 아키텍처와 개발 방식이 코드 기준으로 설명되도록 만든다.

---

## 1. 프로젝트 한 줄 소개

Pocketflow는 1인 가구의 지출 기록, 예정 지출, 예산 관리, 그리고 구매 판단을 하나의 흐름으로 연결한 개인 가계부 서비스입니다.

핵심은 단순 기록 앱이 아니라, 다음 세 가지를 연결하는 데 있습니다.
- 오늘의 지출 기록
- 앞으로 나올 예정 지출과 현금흐름 예측
- AI 기반 구매 판단

즉, 사용자가 “지금 사도 되는가?”를 감으로 판단하지 않고, 이미 입력된 지출 계획과 현금흐름을 바탕으로 판단할 수 있게 만드는 서비스입니다.

---

## 2. 영상에서 보여줄 데모 흐름

발표 영상은 아래 순서로 진행하면 가장 자연스럽습니다.

### 2-1. 시작 화면: 홈 대시보드

먼저 메인 화면인 홈에서 시작합니다.

- 상단 KPI 카드에서 현재 잔액, 30일 예상 유입/유출, 월말 예상 잔액을 보여줍니다.
- 중앙에는 `Recharts` 기반의 30일 현금흐름 그래프가 있습니다.
- 오른쪽에는 AI 구매 판단 패널이 있어, 사용자가 물건 이름과 가격, 이유를 입력하면 즉시 판단 결과를 받을 수 있습니다.

이 화면의 의미는 “이 앱은 단순 가계부가 아니라, 예측과 판단까지 연결된 현금흐름 대시보드”라는 점을 보여주는 것입니다.

### 2-2. 가계부 기록 화면

다음으로 `/expense-log` 화면으로 이동합니다.

이 화면에서는 세 가지가 이어집니다.
- 실제 지출 입력
- 최근 지출 내역 확인
- 예정된 지출 확인과 결제 완료 처리

여기서 중요한 데모 포인트는 예정된 지출 항목이 도래했을 때, 단순히 목록으로만 보는 것이 아니라 결제 완료를 누르면 실제 기록으로 들어가고, 동시에 다음 예정일이 갱신된다는 점입니다.

최근에 추가한 기능 기준으로는 이 화면에서 예정 지출 금액을 직접 수정할 수 있고, 수정한 값이 개별 생필품 내역과 동기화됩니다.

### 2-3. 생필품 설정 화면

마지막으로 `/cashflow-setup` 화면으로 이동합니다.

이 화면은 정기지출과 반복 수입, 개별 생필품을 설계하는 곳입니다.

보여줄 포인트는 다음과 같습니다.
- 개별 생필품 목록이 카드 형태로 정리되어 있음
- 우측 상세 패널에서 항목의 이름, 금액, 주기, 다음 결제일, 메모, 스마트 가격 여부를 수정할 수 있음
- 검색 API를 통해 예상 가격을 받아와 금액에 반영할 수 있음

이 화면은 “미래의 지출 구조를 먼저 설계하고, 그 결과가 홈과 가계부 기록에 반영된다”는 구조를 보여주는 데 중요합니다.

## 3. 아키텍처 개요

프로젝트는 Next.js App Router를 기반으로 하고, 기능 단위로 나눈 구조를 사용합니다.

### 3-1. 전체 레이어

코드 흐름은 다음처럼 이해하면 됩니다.

```text
src/app
-> src/features
-> src/store
-> src/shared
```

각 레이어 역할은 아래와 같습니다.

- `src/app`
  - 페이지 진입점, 라우팅, API route를 담당합니다.
  - 실제 화면은 대부분 feature shell을 불러오는 얇은 엔트리입니다.
- `src/features`
  - 화면별 UI와 행동 로직을 담당합니다.
  - 예: 홈 대시보드, 가계부 기록, 생필품 설정, AI 구매 시뮬레이터.
- `src/store`
  - 전역 상태와 파생 계산을 담당합니다.
  - 정기 지출, 반복 수입, 생필품, 예정 지출, 예산 요약을 한 곳에서 관리합니다.
- `src/shared`
  - 공통 레이아웃, 네비게이션, 유틸리티, 공통 UI 설정을 담당합니다.

### 3-2. 핵심 기술 스택

실제 코드 기준 기술 스택은 다음과 같습니다.

- `Next.js 16`
- `React 19`
- `TypeScript`
- `Tailwind CSS 4`
- `Zustand`
- `Recharts`
- `lucide-react`
- `next-themes`

외부 API는 두 개를 사용합니다.

- `Naver Shopping API`
  - 개별 생필품의 가격 검색과 평균/추천 가격 계산에 사용
- `Gemini API`
  - AI 구매 판단 텍스트 생성에 사용

### 3-3. 공통 레이아웃

`src/shared/layout/app-shell.tsx`가 모든 페이지를 감싸는 공통 레이아웃입니다.

- 좌측 사이드바
- 상단 헤더
- 본문 영역
- 부트스트랩 컴포넌트

이 구조 덕분에 모든 페이지가 같은 내비게이션과 같은 상태 동기화 규칙을 공유합니다.

또한 `CashflowBootstrap`이 앱 시작 시점과 브라우저 포커스/탭 복귀 시점에 예정 항목을 동기화합니다.

즉, 날짜가 바뀌었는데 사용자가 탭을 오래 열어 둔 경우에도 예정 지출 상태가 자동으로 갱신됩니다.

## 5. 상태 관리 아키텍처

### 5-1. Zustand를 쓴 이유

이 프로젝트는 여러 화면이 하나의 공통 지출 데이터를 공유합니다.

예를 들면 아래와 같습니다.

- 생필품 설정 화면에서 금액을 수정하면
- 가계부 기록 화면의 예정 지출이 같이 바뀌어야 하고
- 홈 대시보드의 KPI와 현금흐름 그래프도 다시 계산되어야 합니다

이런 구조에서는 각 화면이 제각각 로컬 상태를 가지면 동기화가 어려워집니다.

그래서 `src/store/cashflow-store.ts`를 단일 원본으로 두고, 나머지 화면은 이 store를 구독만 하게 만들었습니다.

### 5-2. store가 관리하는 데이터

store에는 다음 데이터가 들어 있습니다.

- `fixedExpenses`
- `recurringIncomes`
- `semiFixedExpenses`
- `upcomingExpenses`
- `summary`
- `cashflowSeries`
- `upcomingImpactRows`
- `selectedSemiFixedExpenseId`
- `lastCheckedDate`

즉, 입력 데이터와 계산 결과를 같이 들고 있습니다.

### 5-3. 파생 계산 구조

store는 원시 데이터만 저장하지 않고, 즉시 쓸 수 있는 파생 결과도 생성합니다.

대표적인 계산 함수는 아래와 같습니다.

- `buildCashflowSummary`
- `buildCashflowSeries`
- `buildForecastCashflowSeries`
- `buildUpcomingImpactRows`

이렇게 한 이유는 화면 컴포넌트가 매번 같은 계산을 중복 수행하지 않게 하기 위해서입니다.

예를 들어:

- 홈에서는 30일 현금흐름 그래프가 필요하고
- 가계부 기록에서는 예정 지출 합계가 필요하며
- AI 판단 패널에서는 월말 예상 잔액과 남은 여유 현금이 필요합니다

이 계산을 store에서 공통화해 두면 화면은 표현에만 집중할 수 있습니다.

### 5-4. 날짜 동기화

`CashflowBootstrap`은 앱 시작 시점과 포커스 복귀 시점에 `syncScheduledItems`를 실행합니다.

이 동작은 다음 문제를 해결합니다.

- 사용자가 며칠 동안 앱을 열어두는 경우
- 예정 지출의 다음 결제일이 지나버리는 경우

즉, 사용자가 다시 돌아왔을 때 화면이 현재 날짜 기준으로 다시 정렬되도록 만듭니다.

## 6. 데이터 흐름

이 서비스는 데이터 흐름이 꽤 명확합니다.

### 6-1. 입력 흐름

```text
사용자 입력
-> 폼 검증
-> Zustand store 반영
-> 파생 계산 갱신
-> 화면 리렌더
```

### 6-2. 예정 지출 결제 흐름

```text
예정 지출 도래
-> ScheduledExpenseReviewPanel 표시
-> 결제 완료 클릭
-> confirmSemiFixedExpense 실행
-> 금액/다음 결제일 갱신
-> 가계부 기록에 실제 지출 반영
-> 홈 KPI와 그래프 재계산
```

### 6-3. 생필품 금액 수정 흐름

```text
생필품 상세 또는 예정 지출 화면에서 금액 수정
-> updateSemiFixedExpense 실행
-> semiFixedExpenses 배열 갱신
-> summary / chart / forecast 재계산
-> 다른 화면에서도 같은 금액으로 표시
```

이 구조 때문에 “한 화면에서 바꾸면 다른 화면에도 바로 반영되는” 경험이 가능합니다.

### 6-4. AI 구매 판단 흐름

```text
물건 이름 + 가격 + 사유 입력
-> 현재 현금흐름 컨텍스트 계산
-> /api/ai-purchase-simulator 호출
-> Gemini가 JSON 형식의 판단 결과 생성
-> verdict / advice / judgeNote 렌더
-> 구매 또는 보류 후속 행동 가능
```

## 7. 외부 API 구조

### 7-1. Naver Shopping API

`src/app/api/shopping-price-search/route.ts`는 쇼핑 검색 결과를 받아와 가격을 정리하는 서버 라우트입니다.

이 라우트는 단순 중계가 아니라 다음 일을 추가로 수행합니다.

- HTML 태그 제거
- 가격 파싱
- 이상치 제거
- 최저/중간/평균/추천 가격 계산
- 5분 캐시 적용

이 결과는 `DetailSettingsPanel`에서 개별 생필품의 가격 추천에 사용됩니다.

### 7-2. Gemini API

`src/app/api/ai-purchase-simulator/route.ts`는 AI 구매 판단을 생성하는 서버 라우트입니다.

프롬프트에는 아래 정보가 들어갑니다.

- 물건 이름
- 가격
- 현재 사용 가능 현금
- 사용 이유
- 퍼소나
- 월말 예상 잔액
- 남은 일수
- 고정비/예정비 비중

즉, AI는 단순히 “좋다/나쁘다”를 말하는 게 아니라, 현금흐름 맥락을 포함해서 판단합니다.

그리고 응답은 반드시 JSON 구조로 받도록 강제합니다.

이 덕분에 프론트에서는 AI 결과를 예측 가능하게 렌더링할 수 있습니다.

## 8. 제가 개발한 workflow / 방식

이 프로젝트의 실제 개발 방식은 “Issue 중심, 기능 단위, store 우선 설계”입니다.

### 8-1. 작업 시작 방식

새 기능을 만들 때는 먼저 `issues_backlog.md` 또는 GitHub Issue 기준으로 작업 범위를 고정했습니다.

이 방식의 장점은 다음과 같습니다.

- 이번 작업의 범위를 벗어나지 않게 된다
- UI, store, API를 분리해서 생각할 수 있다
- 완료 기준이 명확해진다

### 8-2. 구현 순서

보통 아래 순서로 개발했습니다.

1. 먼저 관련 문서와 기존 구현을 확인한다.
2. 데이터가 어디에서 생성되고 어디에서 소비되는지 찾는다.
3. store에 단일 원본이 필요한지 판단한다.
4. 화면 컴포넌트를 기능 단위로 분리한다.
5. 서버 라우트가 필요한 경우 API를 만든다.
6. 타입체크로 끊긴 부분을 확인한다.
7. 커밋하고 푸시한다.

이 순서는 실제로 이번 프로젝트에서 반복적으로 쓴 방식입니다.

### 8-3. 이 프로젝트에서 특히 중요했던 원칙

#### 단일 원본 유지

생필품 금액처럼 여러 화면이 공유하는 값은 한 곳에서만 바뀌어야 합니다.

그래서 예정 지출 화면에서 금액을 수정하면 store의 `semiFixedExpenses`가 바뀌고, 개별 생필품 설정 화면도 같은 값을 읽게 만들었습니다.

#### 파생 데이터는 계산해서 쓰기

카드, 그래프, AI 판단용 수치는 각각 필요가 다릅니다.

그래서 입력값과 파생값을 분리했습니다.

#### UI는 얇게, 로직은 가까이

`src/app`은 페이지 조립만 하고, 실제 로직은 `src/features`와 `src/store`에 넣었습니다.

이렇게 하면 화면이 추가돼도 페이지 파일이 복잡해지지 않습니다.

#### 서버에서 해야 할 일은 서버에서 처리

쇼핑 검색과 AI 판단은 외부 API와 키를 사용하므로 서버 라우트로 분리했습니다.

이 방식은 보안 측면에서도 낫고, 프론트는 결과만 소비하면 됩니다.

## 9. 발표에서 강조하면 좋은 기술 포인트

발표 중에는 아래 포인트를 강조하면 좋습니다.

- 단순 가계부가 아니라 “예측형 지출 관리” 서비스라는 점
- 홈 / 기록 / 설정이 전역 상태 하나로 연결돼 있다는 점
- 예정 지출이 도래하면 실제 지출 기록으로 자연스럽게 이어진다는 점
- 생필품 금액 수정이 양쪽 화면에 동시에 반영된다는 점
- AI 판단이 현재 잔액이 아니라 월말 예측과 예정비까지 고려한다는 점

---

## 10. 화면별 말하기 스크립트 초안

영상에서 그대로 읽어도 되는 수준으로 적어두면 아래와 같습니다.

### 10-1. 오프닝

“Pocketflow는 1인 가구의 지출을 기록하는 데서 끝나지 않고, 앞으로 나올 지출까지 예측해서 지금 사도 되는지를 판단해 주는 개인 가계부 서비스입니다.”

### 10-2. 홈 설명

“첫 화면에서는 현재 잔액과 30일 예상 유입, 유출, 월말 예상 잔액을 한눈에 볼 수 있습니다. 가운데 그래프는 날짜별 현금흐름을 보여주고, 오른쪽 AI 패널은 물건 이름과 가격을 넣으면 구매 판단을 내려줍니다.”

### 10-3. 가계부 기록 설명

“가계부 기록 화면에서는 실제 지출을 입력할 수 있고, 이미 예정된 지출은 별도로 확인할 수 있습니다. 도래한 항목은 결제 완료로 처리하면 실제 지출 기록으로 반영되고, 다음 결제일도 자동으로 넘어갑니다.”

### 10-4. 최근 추가한 수정 기능 설명

“최근에 추가한 기능은 예정 지출 화면에서 금액을 바로 수정하는 기능입니다. 여기서 바꾸면 같은 store를 공유하는 개별 생필품 내역에도 자동으로 반영됩니다. 즉, 한쪽만 바뀌는 게 아니라 데이터 원본이 같이 바뀝니다.”

### 10-5. 생필품 설정 설명

“생필품 설정 화면은 미래 지출을 설계하는 곳입니다. 항목별 금액, 주기, 다음 결제일, 메모를 관리하고, 쇼핑 검색으로 가격 추천도 받을 수 있습니다.”

### 10-6. 아키텍처 설명

“구조는 Next.js App Router 위에 feature 단위 UI를 두고, Zustand store가 공통 상태를 담당하며, 외부 API 호출은 서버 route로 분리했습니다. 이렇게 하면 화면은 표현에 집중하고, 계산과 동기화는 store에서 일관되게 처리할 수 있습니다.”

### 10-7. 개발 방식 설명

“개발할 때는 Issue를 먼저 기준으로 잡고, 관련 문서와 코드를 확인한 뒤 store와 화면, API를 순서대로 분리해서 구현했습니다. 각 작업 뒤에는 typecheck로 깨진 부분을 확인하고, 커밋과 푸시까지 한 번에 마무리했습니다.”

## 11. 제출 전 체크리스트

- 영상은 반드시 음성 설명이 들어가야 한다
- 화면 녹화 방식으로 촬영한다
- 서비스 데모가 들어가야 한다
- 아키텍처 설명이 들어가야 한다
- 내가 만든 workflow 또는 개발 방식 설명이 들어가야 한다
- 가능하면 5분 내외로 압축한다

## 13. 결론

Pocketflow의 핵심은 “기록”이 아니라 “흐름”입니다.

현재 잔액, 예정 지출, 반복 지출, AI 판단, 그리고 실제 결제 기록을 하나의 데이터 흐름으로 묶어서 사용자가 더 현실적으로 지출을 결정하게 만드는 것이 이 프로젝트의 목적입니다.

발표에서는 이 점을 중심으로 설명하면, 단순 기능 나열이 아니라 문제 해결형 서비스라는 인상을 줄 수 있습니다.
