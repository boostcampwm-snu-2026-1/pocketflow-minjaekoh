# Pocketflow 개발 워크플로우

## 1. 목적
Pocketflow는 개인의 지출 기록, 정기 지출 관리, 현금흐름 예측을 돕는 재무 관리 대시보드입니다.

이 문서는 프로젝트를 구현하기 전에 개발 흐름을 정리하기 위한 워크플로우 문서입니다. 과제 요구사항에 맞게 Agent를 활용한 개발 과정을 체계화하고, 3주 동안 지속 가능한 구조로 프로젝트를 발전시키는 것을 목표로 합니다.

## 2. 워크플로우 원칙
- 기능을 바로 구현하기 전에 화면, 데이터, 모듈 경계를 먼저 정리합니다.
- AI가 생성한 코드는 최종 구현물이 아니라 참고용 초안으로 다룹니다.
- 폴더명과 파일명만 보고도 기능과 책임을 이해할 수 있게 구성합니다.
- 단일책임원칙을 기준으로 파일을 분리합니다.
- 얇게 쪼개진 shallow module보다 내부 책임이 명확한 deep module을 지향합니다.
- 문서와 코드 파일은 UTF-8 인코딩을 기준으로 관리합니다.

## 3. 전체 개발 흐름
1. `PRD.md`를 기준으로 프로젝트 범위와 핵심 기능을 확정합니다.
2. `issues_backlog.md`를 기준으로 구현 순서를 정리합니다.
3. UI 참고 이미지와 Vercel AI 생성 결과물을 검토합니다.
4. 컨셉맵, 의존성맵, 데이터 흐름을 간단히 정리합니다.
5. 폴더 구조와 모듈 경계를 확정합니다.
6. 공통 레이아웃과 라우팅을 먼저 구현합니다.
7. 화면 단위로 기능을 구현합니다.
8. Zustand 상태 관리와 계산 로직을 연결합니다.
9. AI 분석 흐름을 연결하고 응답 검증 방식을 정리합니다.
10. 최종적으로 문서, 화면, 동작 흐름을 검토합니다.

## 4. 브랜치 운영 방식
기본 개발 흐름은 `main -> dev -> feature` 구조를 따릅니다.

- `main`: 최종 안정 버전만 유지합니다.
- `dev`: 기능 브랜치들이 합쳐지는 개발 기준 브랜치입니다.
- `feature/*`: 이슈별 실제 작업 브랜치입니다.

작업 규칙은 다음과 같습니다.

- 각 이슈 작업을 시작할 때 Codex가 새 feature 브랜치를 만들고 해당 브랜치로 이동합니다.
- feature 브랜치는 항상 `dev`에서 분기합니다.
- 브랜치 이름은 `feature/issue-번호-작업요약` 형식을 사용합니다.
- 예시: `feature/issue-2-layout-routing`
- 작업이 끝나면 Codex는 커밋하지 않습니다.
- 커밋은 사용자가 직접 수행합니다.
- Codex는 작업 종료 시 한국어 커밋 메시지를 추천합니다.
- Codex는 작업 종료 시 GitHub PR description 초안도 함께 제공합니다.
- Codex는 이슈 작업이 끝날 때 `issues_backlog.md`의 해당 이슈 체크리스트를 완료 상태로 변경합니다.
- 사용자는 feature 작업이 끝날 때마다 해당 브랜치를 `dev`로 PR 생성 후 병합합니다.
- 다음 이슈는 이전 feature 브랜치가 `dev`에 병합된 뒤 시작합니다.
- Codex는 새 이슈를 시작하기 전에 `dev`가 최신인지 확인하고, 이전 이슈 변경이 `dev`에 없으면 작업을 시작하지 않습니다.

추천 명령 흐름은 다음과 같습니다.

```powershell
git checkout dev
git pull origin dev
git checkout -b feature/issue-2-layout-routing
```

작업 완료 후 사용자가 커밋할 때는 다음 흐름을 사용합니다.

```powershell
git status
git add .
git commit -m "feat: 공통 레이아웃과 라우팅 뼈대 구현"
git push -u origin feature/issue-2-layout-routing
```

이후 GitHub에서 `feature/issue-2-layout-routing -> dev` 방향으로 PR을 생성하고 병합합니다.

다음 이슈를 시작할 때는 반드시 병합된 `dev`에서 새 브랜치를 생성합니다.

```powershell
git checkout dev
git pull origin dev
git checkout -b feature/issue-3-kpi-cards
```

## 5. 컨셉맵
프로젝트는 다음 4개 영역을 중심으로 나눕니다.

```text
pocketflow
├─ home
├─ expense-log
├─ cashflow-setup
└─ ai-workflow
```

- `home`: 현금흐름 예측, KPI 카드, 다가오는 지출, AI 소비 분석 요약
- `expense-log`: 신규 지출 입력, 최근 지출 내역, AI 소비 시뮬레이터
- `cashflow-setup`: 고정비, 준고정비, 상세 설정 패널
- `ai-workflow`: 입력 정규화, 프롬프트 구성, AI 요청, 응답 검증

## 6. 의존성맵
의존성은 아래 방향으로만 흐르게 합니다.

```text
app
└─ features
   └─ entities
      └─ shared
```

- `app`: 페이지 조립과 라우팅을 담당합니다.
- `features`: 화면별 기능과 사용자 상호작용을 담당합니다.
- `entities`: 거래, 정기 지출 같은 핵심 도메인 데이터를 담당합니다.
- `shared`: 공통 UI, 유틸 함수, 설정 값을 담당합니다.

화면에서 바로 계산 로직을 처리하지 않고, feature 또는 entity 쪽으로 책임을 분리합니다.

## 7. 데이터 흐름
Pocketflow의 핵심 데이터 흐름은 다음과 같습니다.

```text
사용자 입력
-> 데이터 정규화
-> 상태 저장
-> 현금흐름 계산
-> AI 분석 요청
-> 응답 검증
-> 화면 반영
```

예를 들어 지출을 추가하면 다음 흐름을 따릅니다.

```text
지출 입력
-> 금액, 날짜, 카테고리 정규화
-> Zustand Store 반영
-> 예상 월말 잔액 재계산
-> 차트 데이터 갱신
-> 대시보드 UI 업데이트
```

AI 소비 시뮬레이터는 다음 흐름을 따릅니다.

```text
구매 희망 항목 입력
-> 현재 잔액과 예정 지출 컨텍스트 생성
-> AI 요청
-> JSON 또는 구조화된 응답 검증
-> 위험도와 설명을 UI에 표시
```

## 8. Agent 활용 방식
Agent에게 작업을 요청할 때는 다음 정보를 함께 제공합니다.

- 현재 작업할 Issue 번호
- 관련 화면 또는 컴포넌트
- 참고해야 할 문서
- 기대 결과
- 직접 검증해야 할 체크포인트

요청 예시는 다음과 같습니다.

```text
Issue #7 신규 지출 입력 폼을 구현해줘.
참고 화면은 지출관리 페이지 이미지이고, 날짜/항목명/금액/카테고리 입력이 필요해.
폼 입력 후에는 로컬 상태에 반영되는 구조로 만들어줘.
```

Agent가 만든 결과물은 바로 확정하지 않고 다음 기준으로 검토합니다.

- 이슈 요구사항을 충족하는가
- 기존 폴더 구조와 맞는가
- 불필요하게 얇은 파일로 쪼개지지 않았는가
- 상태와 계산 로직이 UI에 과하게 섞이지 않았는가
- 한글 문서나 UI 텍스트가 깨지지 않는가

## 9. Vercel AI 결과물 처리 기준
Vercel AI로 생성한 화면 코드는 실제 구현의 기준이 아니라 참고 자료로 사용합니다.

- 화면 분위기와 레이아웃을 확인하는 용도로 사용합니다.
- 좋은 컴포넌트 구조나 스타일은 선별해서 가져옵니다.
- 상태 관리, 계산 로직, AI 요청 흐름은 프로젝트 구조에 맞게 다시 정리합니다.
- 생성 코드는 실제 `src` 구조와 분리된 참고 폴더에 보관합니다.

## 10. 문서 및 인코딩 규칙
- 모든 Markdown 문서는 UTF-8로 저장합니다.
- `PRD.md`, `issues_backlog.md`, `WORKFLOW.md`, `README.md`의 용어를 최대한 통일합니다.
- 한글 문서를 수정한 뒤에는 GitHub 또는 에디터에서 깨짐 여부를 확인합니다.
- 터미널 출력이 깨지더라도 파일 자체가 UTF-8인지 별도로 확인합니다.

## 11. 1주차 목표
1주차에는 기능 구현보다 프로젝트 기획과 워크플로우 정리에 집중합니다.

- 프로젝트 기획서 초안 정리
- 전체 이슈 백로그 정리
- UI 참고 이미지 분석
- Vercel AI 결과물 보관 방식 결정
- Agent 개발 워크플로우 초안 작성
- 인코딩 및 문서 관리 규칙 정리

## 12. 이후 진행 방향
- 2주차에는 초기 세팅, 공통 레이아웃, 핵심 화면 구현을 진행합니다.
- 3주차에는 상태 관리, 계산 로직, AI 분석 흐름을 연결합니다.
- 마지막에는 전체 화면 흐름과 문서 산출물을 함께 검토합니다.
